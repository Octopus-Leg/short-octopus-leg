# 4장 독자 관점에서 릴리스 문서와 장애 보고서 쓰기

## 01 체인지 로그를 분류, 요약, 종합하는 법

### 체인지 로그의 양과 만족도의 관계

- 체인지 로그는 너무 짧아도, 너무 길어도 좋지 않으며, **읽기 좋은 적절한 양**으로 써야 상사와 고객 모두 만족함.
- 적절한 양으로 쓰려면 선정 → 분류 → 요약 → 종합의 4단계 과정을 거쳐야 함.

### 1단계: 선정하기

- 개발자가 노력을 많이 들인 것 기준이 아닌, **회사·개발자·독자(고객)의 관심**을 기준으로 체인지 로그를 선정해야 함.
- 회사·개발자·독자의 관심을 조합한 8가지 경우로 우선순위를 분류할 수 있음.

| 회사가 말하고 싶다 | 개발자가 말하고 싶다 | 독자가 듣고 싶다 | 순위 | 수준 | 예시 |
|:---:|:---:|:---:|:---:|:---:|:---:|
| O | O | O | 1순위 | 자세히 | 인공지능 추천 추가 |
| O | O | X | 2순위 | 간단히 | 대용량 네트워크 증설 |
| O | X | O | 3순위 | 간단히 | UI 개선 |
| O | X | X | 4순위 | 주석 | 면책 조항, 정책 변경 |
| X | O | O | 5순위 | 참고 | 작업 과정, 레퍼런스 |
| X | O | X | 삭제 | - | 성과가 낮은 오류 수정 |
| X | X | O | 삭제 | - | 경쟁력 없는 기능 추가 |
| X | X | X | 삭제 | - | 사소한 오류 수정 |

### 2단계: 분류하기

- 공개할 체인지 로그를 선정했으면 비슷한 것끼리 묶어 분류함.
- **개발 관점** 분류: 독자가 개발자인 경우에 유용하며, 새로운 기능 추가 / 기능 개선 / 오류 수정으로 묶음.
- **사용자 관점** 분류: 독자가 일반 사용자인 경우에 유용하며, 사용자가 서비스를 이용하는 각 단계(예: 게임 준비 / 게임 중 / 게임 종료)에 따라 묶음.

### 3단계: 요약하기

- 서술식 문장을 **개조식 문장**으로 바꾸면 내용이 자연스럽게 축약되고 의미가 더 분명해짐.
- 개조식으로 만들려면 불필요한 부사나 형용사, 조사나 어미를 없애고 정확하고 적절한 단어로 대체함.
- 개조식으로 변환 후 **마침표를 제거**하여 최종 정리함.

### 4단계: 종합하기

- 릴리스 내용 전체를 **한 문장으로 만들어 체인지 로그의 첫 줄**에 적어야 함.
- 종합 문장은 분석과 반대 방향으로, 개념화하지 않고 **특성이나 결과, 또는 특성과 결과를 합쳐** 서술함.
  - 예) "토끼는 동물이다"(분석) → "토끼는 귀가 밝아서 잘 도망간다"(종합)
- 내용이 두 가지 이상이면 줄을 나누고, 여러 특성과 결과를 분석해 **사용성·안정성·편리성·효율성 같은 추상적 상위 개념**을 제목으로 사용함.
- 종합 문장은 다음 두 질문에 답할 수 있어야 함.
  - "그래서, 한마디로 뭘 했다는 거죠?" → 상위 개념으로 답변
  - "그래서, 핵심이 뭔가요?" → 종합 개조식 문장으로 답변

## 02 고객에게 유용한 정보를 쓰자

### 개발자 관점과 고객 관점

- 외부 개발자나 일반 사용자가 보는 체인지 로그는 **개발자 관점이 아닌 고객 관점**으로 써야 함.
- 고객 관점으로 쓰려면 **변경사항이 고객에게 끼치는 영향(고객의 문제 해결)** 을 체인지 로그에 추가해 기술함.
- 고객의 문제 해결을 앞쪽에 배치하고, 개발자의 문제는 짧게 줄여 뒤에 괄호로 표기하는 방식이 효과적임.
  - 예) "이제 애니메이션 스티커를 정상적으로 댓글에 사용할 수 있습니다(화면 멈춤 문제 해결)."
- 내용에 따라 관점을 적절히 선택하며, 정말 소소한 것은 마지막 문장에 **'기타'로 묶는 것**도 좋은 방법임.

### 과거를 리뷰하고 미래를 보여주자

- 체인지 로그에는 이번 릴리스의 내용뿐만 아니라 **다음 릴리스에서 검토 중인 항목**도 공개하는 것이 좋음.
- 이전 릴리스에서 잡은 버그가 완벽히 해결됐는지, 새 기능은 어떤 것을 많이 사용하는지도 함께 공유함.
- 예) 카카오맵 체인지 로그 형식: 【주요 개선 항목】 다음에 【검토 중인 항목】 섹션을 추가함.

### Semantic Versioning(유의적 버전)

- 버전은 `X.Y.Z` 형태로 표기하며, X(Major) / Y(Minor) / Z(Patch)로 구분함.
- 고객은 버전을 보고 체인지의 범위와 중요성을 먼저 판단하므로, **버전 번호 규칙을 일관성 있게 지키는 것**이 중요함.

| 자리 | 올리는 시점 | 의미 |
|:---:|:---:|:---:|
| X (Major) | 기존 버전과 호환되지 않는 변화 시 | 전면 업그레이드, 이전 버전과 거의 호환 안 됨 |
| Y (Minor) | 기존 버전과 호환되는 새로운 기능 추가 시 | 새 기능 추가, 1.2.x와 일부 호환 가능 |
| Z (Patch) | 기존 버전과 호환되는 작은 규모의 패치 시 | 간단한 패치, 이전 버전(1.2.x)과 호환 |

- X를 올리면 Y는 0으로 초기화되고, X나 Y를 올리면 Z는 0으로 초기화됨.
- 초기 내부 개발 버전은 `0.1.0`으로 시작하며, 최초 공개 API는 `1.0.0`이어야 함.
- 정식 배포 전 사전 배포(pre-release)가 필요한 경우 Z 다음에 붙임표(-)로 구분하고 식별자를 적음.
  - 예) 알파 버전: `1.0.0-alpha`, 알파 버전이 여러 개: `1.0.0-alpha.1`
- 일단 배포된 버전은 변경해서는 안 되며, 변경이 있다면 무조건 버전을 올려야 함.

## 03 릴리스 문서는 문제 해결 보고서처럼 쓰자

### 문제와 문제점을 구별하자

- **문제**: 목표와 현상의 차이로, 발생형 / 탐색형 / 설정형의 세 가지 종류가 있음.
  - **발생형**: 기준에서 이탈해 원상 복구가 필요한 문제 (예: 프로그램 에러)
  - **탐색형**: 현재 상황을 개선하거나 효율을 높이는 문제 (예: 프로그램 개선, 시스템 효율화)
  - **설정형**: 새로운 과제나 목표를 설정함으로써 일어나는 미래 지향적 문제 (예: 새로운 기능, 대폭 업그레이드)
- **문제점**: 문제를 발생시키는 여러 현상들로, 문제와 문제점을 반드시 구별해서 써야 함.

### 문제, 문제점, 해결책, 후속 계획 순으로 적자

- 릴리스 문서는 **문제 → 문제점 → 해결책 → 후속 계획** 순서로 작성해야 고객이 내용을 정확히 파악하고 향후 변경 계획도 짐작할 수 있음.
- 하나의 문제에 문제점은 여러 가지이며, 예산과 인력의 한계로 특정 문제점을 선택해 해결할 수밖에 없으므로 **어떤 문제점을 선택했는지** 독자에게 명확히 알려줘야 함.

### 법적인 문제를 고려해서 쓰자

- 릴리스 노트는 계약에 의한 산출물로 취급될 수 있으므로 **면책 조항**을 반드시 포함하는 것이 좋음.
- 독자가 반드시 해야 할 일, 우선순위가 있는 방법, 자유롭게 선택 가능한 것은 서술어로 구분해 표현함.

| 구분 | 서술어 표현 예시 |
|:---:|:---|
| **필수** | "~해야 한다", "~하지 않으면 안 된다", "~하면 안 된다", "~해서는 안 된다" |
| **권장** | "~할 것을 권장한다", "~하는 것이 좋다", "~하는 것이 이상적이다" |
| **선택** | "~할 수도 있다", "~해도 된다", "~하는 방법이 있다" |

- 면책 조항이 있어도 **서술어를 잘못 쓰면 질타**를 크게 받을 수 있으므로 필수/권장/선택의 구분을 명확히 해야 함.
- 면책은 면책으로만 받아들이지 말고, **다음 릴리스 노트의 해결 대상**으로 적어 개선 의지를 보여주는 것이 효과적임.

## 04 비즈니스를 이해하는 장애 보고서 쓰기

### 장애 보고서의 특징

장애 보고서에는 일반 개발 문서와 다른 4가지 특징이 있음.

1. **장애 보고서는 개발자가 원할 때 쓸 수 없음**: 장애 인지 순간부터 해결 직후 또는 다음날 오전까지가 작성 시간이므로, 무엇보다 **신속한 글쓰기**가 필요함.
2. **장애의 1차 원인은 대부분 다른 원인의 결과임**: 최초 원인을 찾으려면 원인의 원인을 계속 추적해야 하며, 재발을 막으려면 **원인뿐만 아니라 이유(사람이 주어)** 까지 찾아야 함.
3. **장애 보고를 받는 윗사람은 대부분 개발자가 아님**: 특히 인문계 출신 임원은 장애를 비즈니스에 주는 영향으로 보므로, **비즈니스 관점의 글쓰기**가 필요함.
4. **장애를 해결했다고 해서 100% 해결한 것은 아님**: 윗사람은 확정적인 대답을 원하므로, 확정적이고 신뢰할 만한 결단을 정치적으로 내려야 하는 **정치적 글쓰기**가 필요함.

### 질문에 대답하는 신속한 글쓰기

- 장애를 아주 단순한 주어·목적어·서술어 문장으로 먼저 쓴 뒤, **동료가 물어볼 질문에 말로 대답하듯** 작성하면 신속하게 글을 완성할 수 있음.
- 동료의 질문은 곧 장애 보고서의 각 항목임.

| 동료의 질문 | 장애 보고서 항목 |
|:---|:---:|
| "뭐야, 뭐야? 무슨 일이야?" | 장애 내용 |
| "그래서 어떻게 됐어?" | 장애 영향 |
| "왜 그렇게 됐어?" | 장애 원인 |
| "넌 어떻게 했는데?" | 조치 사항 |
| "그럼 이제 장애는 해결된 거야?" | 조치 결과 |
| "누가 잘못한 거야?" | 핵심 원인 |
| "그럼 이제 어떻게 할 거야?" | 항후 대책 |

### 원인과 이유를 찾는 분석적 글쓰기

- **5Whys 기법**: "왜?"를 다섯 번 반복해 문제의 근본 원인을 찾는 기법으로, 피상적 원인이 아닌 근본 원인을 발견하는 데 사용함.
- 5Whys에서 **'why'의 의미는 두 가지**로 이해해야 함.
  - **원인**: 사물이나 현상, 동작이 문제를 초래하는 일이나 사건
  - **이유**: 사람이 어떤 일을 하거나 하지 않은 까닭이나 동기
- 재발을 원천적으로 막으려면 원인만 해결해서는 안 되며, **그 원인이 발생한 이유(사람이 주어가 되는 이유)** 까지 찾아 해결해야 함.

### 상사를 고려하는 비즈니스 관점의 글쓰기

- 장애 보고서를 비즈니스 관점으로 쓴다는 것은 장애의 영향을 **매출과 비용으로 환산해 설명**하는 것임.
- 장애로 인한 손실 계산 순서:
  1. 장애 발생 시간대의 **기대 매출** 계산 (최근 1개월 같은 요일·시간대 매출 평균)
  2. 장애 때 **실제 발생 매출** 계산
  3. **매출 손실** = 기대 매출 - 실제 매출
  4. 장애 직후 특정 시간까지 매출이 급증했다면 **지연 매출**로 추가 주장 가능
  5. 지연 매출까지 인정하면 **매출 손실** = 기대 매출 - 실제 매출 - 지연 매출
- 결제와 무관한 장애(예: 홈페이지 접속 불가)는 **광고비 손실 + 기대 매출 손실**을 합산해 비즈니스 손실로 환산함.

### 원하는 것을 얻는 정치적 글쓰기

- 장애를 완벽히 해결했다고 보장하기 어렵지만, 윗사람은 **확정적이고 명확한 대답**을 원하므로 정확한 단어와 문장으로 현상과 사실을 있는 그대로 표현해야 함.
- **보카시 장난**(경계를 흐리게 말하는 것)을 피하고, 재발 가능성은 퍼센티지로 명확하게 표현함.

| 재발 가능성 | 표현 |
|:---:|:---|
| 1% | 절대 재발하지 않는다 |
| 10% | 재발하지 않는다 |
| 20% | 재발할지도 모른다 |
| 50% | 재발할 수 있다 |
| 80% | 재발한다 |
| 99% | 반드시 재발한다 |

- 윗사람에게 예산이나 조치를 요청할 때는 위험 상황을 **과격하게 표현**해 예산 확보를 이끌어내야 하는 경우도 있음.
  - 예) "냉방기 미교체 시 화재 발생 염려" → "냉방기 폭발 시 서비스 전면 중단"
- 정치적 글쓰기는 눈치를 보며 얼버무리는 것이 아니라, **정확한 단어와 문장으로 현상과 사실을 있는 그대로 표현**하는 것임.
