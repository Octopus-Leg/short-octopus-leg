# 2장 개발 시간을 줄여주는 이름 짓기와 주석 쓰기

## 01 네이밍 컨벤션, 이유를 알고 쓰자

### 개발자의 가장 큰 고민은 이름 짓기

- 변수, 함수, 클래스, 파일, 디렉토리, DB 칼럼, 프로젝트 이름까지 수많은 이름을 지어야 함.
- 2013년 Quora/Ubuntu 포럼 설문에서 개발자 4,522명 중 절반이 이름 짓기가 가장 힘들다고 대답함.

### 이름 짓기는 창조가 아니라 조합

- 이름 짓기는 무에서 유를 창조하는 것이 아니라, 기존 방식이나 이름을 **차용해서 조합**하는 경우가 대부분임.
- 깃허브 인기 자바 소스 분석 결과("오픈소스의 네이밍 특징들")에서 증명된 주요 규칙:
  - **자바 네이밍 컨벤션을 준수한다**: 클래스는 UpperCamelCase, 함수·변수는 lowerCamelCase, 상수는 UPPER_DELIMITER_CASE
  - **네이밍은 보통 16글자, 3단어를 조합한다**: 클래스 3.18단어, 함수 3.36단어, 변수 2.57단어
  - **품사는 명사·동사·형용사의 조합이다**: 명사+명사+명사, 동사+명사+명사, 형용사+명사+명사 등

### 코드의 네이밍 컨벤션은 영어 표기법을 상속받았다

- 네이밍 컨벤션은 기본적으로 영어 표기법을 준수함.
- 영어 대문자 표기 원칙(Capitalization)의 특성:
  1. 중요하거나 특정한 것을 가리키는 명사는 첫 글자를 대문자로 씀.
  2. 그런 명사들이 이어질 때는 첫 글자를 모두 대문자로 씀.
  3. 명사·관사가 아닌 동사·형용사 등은 소문자를 씀.
- 이 특성이 코딩에 그대로 적용되어 **파스칼 표기법**과 **카멜 표기법**이 만들어짐.

### 파스칼 표기법으로 클래스 이름 짓기

- **파스칼 표기법**: 모든 단어의 첫 글자를 대문자로 씀. 주로 클래스·인터페이스 이름에 사용함.

```java
// 잘못된 예
interface menu
class coffeemenu implements menu

// 좋은 예
interface Menu
class CoffeeMenu implements Menu
```

### 카멜 표기법으로 함수·변수의 이름 짓기

- **카멜 표기법**: 첫 단어를 빼고 나머지 단어의 첫 글자만 대문자로 씀. 주로 함수·변수에 사용함.

```java
// 잘못된 예
int TotalCount = 0;
void OrderCoffee()

// 좋은 예
int totalCount = 0;
void orderCoffee()
```

### 상수는 모두 대문자로 쓴다

- 변수와 구별하기 위해 **모두 대문자로 쓰고 언더스코어(_)로 단어를 연결**함.
- 어느 쪽이든 회사 안에서는 통일하는 것이 좋음.

```java
static final int COFFEE_MAX = 10;
```

### 패키지와 모듈은 모두 소문자로 쓴다

- 패키지·모듈은 클래스를 담아놓은 통에 불과하므로 **소문자로만** 씀.
- 패키지 이름과 클래스 이름의 혼동을 방지하기 위한 목적도 있음.

```java
// 안 좋은 예
kr.co.wikibook.android.DeveloperWriting

// 좋은 예
kr.co.wikibook.android.developerwriting
```

### BEM 표기법

- CSS에서 사용하는 **BEM(Block, Element, Modifier)** 표기법은 **'대상\_\_요소--상태'** 를 의미함.
  - 요소·부분: **언더스코어 두 개(\_\_)**
  - 상태·속성: **하이픈 두 개(--)**

```css
.form {}
.form__button {}
.form__button--disabled {}
```

### 가독성과 소통이 먼저다

- 표기법 자체가 아니라 **그렇게 쓰는 이유**가 중요함. 컨벤션을 만든 이유는 가독성과 소통 때문임.
- 같은 부서나 프로젝트 개발자들끼리는 **코딩하기 전에 기본적인 컨벤션 규칙을 정하는 것이 우선**임.

## 02 변수 이름을 잘 짓는 법

### i는 변수 이름이지만 d는 아니다

- `i`는 integer·index의 첫 글자로 간주되므로 반복문 카운터나 배열 인덱스로 사용해도 무방함.
- `x, y`를 좌표값으로 쓰는 것도 관습으로 박혀 있어 괜찮음.
- `a, d`처럼 아무 의미 없는 글자는 변수명으로 좋지 않음.
  - 임의의 날: `someday` / 오늘: `today` / 특정한 날: `thisDay` / 마지막 날: `finalDay`

### 긴 이름? 짧은 이름? 검색 잘 되는 이름!

- 요즘은 IDE 자동완성 덕분에 **변수 이름의 길이와 오타의 상관관계는 없다**고 봐야 함.
- 변수 타입 앞에 접두어를 붙이는 **헝가리안 표기법**은 IDE 발전으로 거의 레거시(legacy)가 됨.

### 복수형을 나타내는 s를 붙일까 말까?

- `-s` 복수형은 함수명 중간에 쓰면 눈에 잘 띄지 않으므로, 함수명에서는 **'array'나 'list of'** 를 쓰는 편이 더 나을 수도 있음.
- 회사 또는 프로젝트 안에서는 **규칙을 하나로 통일**해야 함.

### 약어를 쓰는 것이 좋을까? 안 쓰는 것이 좋을까?

- 약어는 **보편성을 기준**으로 정하는 것이 좋음. 이미 널리 알려진 용어는 그대로 씀.
  - HTML, UI, UX, temp, doc, param, arg 등
- 약어를 새로 만들고 싶다면 **비개발자에게 물어보는 것**도 좋은 방법임.

### 중요한 단어를 앞에 쓴다

- 변수 이름에서 **본래 의미를 뜻하는 단어를 앞에** 써야 검색하기 쉬움.

```java
// 안 좋은 예
int totalVisitor
int maxSizeOfWindow
int numberOfTotalVIP

// 좋은 예
int visitorTotal
int windowSizeMax
int vipCount
```

### 함수 이름 짓는 순서

1. 함수가 할 일을 **한글 문장으로 작성**함.
2. 사용자 관련 내용을 제거하고 **시스템 동작 중심**으로 남김.
3. 논리 흐름을 확인하여 **함수 개수를 결정**함 (1함수 1업무 규칙).
4. 한글을 **영어로 바꾸고** 정관사·불필요한 단어·소유격을 제거함.
5. of 앞뒤 단어를 바꾸고 **의미상 없어도 되는 단어를 제거**함.
6. **띄어쓰기를 없애고** 두 번째 단어부터 첫 철자를 대문자로 바꿈.

```java
// (함수1)
// 한글: "사용자 이름을 input 태그에서 가져온다"
// 최종: getUserNameFromField()

// (함수2)
// 한글: "사용자 이름의 글자 수가 2글자 이하면 처리한다"
// 최종: checkUserNameUnder2Characters()
```

## 03 좋은 이름의 기준, SMART

### 한 번에 좋은 이름을 지을 수는 없다

- 공기주입기 인형이 Tall Boys → Fly Guys → Tube Man → Sky Dancer로 바뀐 것처럼, 이름은 발전함.

### 좋은 이름이 가진 5가지 특징

**SMART:**
- **easy to Search**: 검색하기 쉽고
- **easy to Mix**: 조합하기 쉽고
- **easy to Agree**: 수긍하기 쉽고
- **easy to Remember**: 기억하기 쉽고
- **easy to Type**: 입력하기 쉽고

### easy to Search: 검색하기 쉽게 이름 짓는 법

- **고전적 범주화**를 이용해 한 단계 상위 범주의 이름을 태그처럼 덧붙이면 됨.

```java
// 안 좋은 예
SERVER_TIMEOUT
NO_RESULT
BAD_REQUEST

// 좋은 예
ERROR_SERVER_TIMEOUT
ERROR_NO_RESULT
ERROR_BAD_REQUEST
```

- 같은 접두어를 가진 변수가 너무 많아지면 오히려 역효과이므로, 먼저 **에러 구분 체계**부터 만들어야 함.

### easy to Mix: 조합하기 쉽게 이름 짓는 법

- 속성(크기, 색깔 등) 대신 **개념으로 이름을 짓는 것**이 좋음. 속성이 바뀌면 이름도 전부 바꿔야 하기 때문임.
- **개발 언어의 문법과 조합**하는 것이 좋음. HTML에서 `title` 하나만 지으면 h1.title, h2.title, p.title처럼 조합해 사용할 수 있음.

### easy to Agree: 수긍하기 쉽게 이름 짓는 법

- 누가 보더라도 **그렇게 짓는 것이 더 낫다고 동의할 수 있는 이름**이어야 함.
- 코드의 유효범위가 좁고 직관적이라면 굳이 길게 이름 지을 필요가 없음.

```java
// 안 좋은 예 (범위가 좁은 루프에서 긴 이름)
for(int indexOfNewEmployeeOfThisMonth = 0; ...)

// 좋은 예
for(int i = 0; i < 9; ++i) { ... }
```

### easy to Remember: 기억하기 쉽게 이름 짓는 법

- **시청각적으로 완결된 단어**가 더 잘 기억됨. 약자를 쓸 때도 시청각적으로 완결시키면 좋음.
  - `BIE`: 바이? 비이? (기억하기 어려움) → `BIOE`: 바이오 이 (연상 가능)
- 개발자만 보는 문서라면 보편적으로 쓰는 이름(HelloWorld, char, AWS, S3 등)은 그대로 써도 무방함.

### easy to Type: 입력하기 쉽게 이름 짓는 법

- 자주 사용되거나 중요한 이름이라면 **오타 가능성**도 고려해야 함.
- 자판으로 입력할 때 잘 틀리는 단어:
  - **연속된 철자**: successes, classes, committee, parallel
  - **묵음**: lambda, thumbnail, debt
  - **ie/ei**: chief, receive, retrieve, friends, achieve
  - **sion/tion**: position, commission
  - **uous/ous/us**: continuous, fabulous, genius

## 04 좋은 코드에는 주석이 없다?

### 이름을 잘 지으면 주석을 줄일 수 있다

- 이름을 잘 지으면 주석을 대폭 줄일 수 있음. **이름이 주석이 할 일을 대신**하기 때문임.

```java
// 안 좋은 예
// 스크린 최대 높이를 480으로 지정함
int h = 480

// 좋은 예
int screenHeightMax = 480
```

### 처음부터 주석 없이 코딩하는 연습을 하자

- 주석 대신 이름으로 의미를 전달할 수 있음.
- 불필요한 주석의 예:
  - `"Ok": true` → 처음부터 `"isRequestSuccess": true`로 지으면 주석이 불필요함.
  - `"success": []` → `"noCreatedBecauseUnsubscriber": []`처럼 이름 자체에 의미를 담음.
- JSON 문법 설명 주석처럼, **코드를 보는 사람이 이미 알 것으로 전제할 수 있는 내용은 주석이 필요 없음**.

### 주석이 필요한 때도 많다

- 개발자마다 영어 실력 차이가 있으므로 어쩔 수 없이 **한글 주석이 필요한 경우**가 있음.
- 한글 주석으로 한 번 더 설명하면 잘못된 영작문이나 실수를 줄일 수 있음. **주석이 코드의 정확성을 높이고 버그를 줄이는 계기**가 됨.
- 주석이 제 역할만 충실하다면 많고 적고는 상관없음.

## 05 다른 개발자를 배려하는 주석 쓰기

### 코드는 의미를, 주석은 의도를

- 코드가 의미를 전달한다면, **주석은 개발자의 의도를 전달**해야 함.
- 코드에 표현하지 못한 의도를 전달해야 할 때 주석을 씀.

```java
// 좋은 예
letsEatSomething() // 내가 배가 고픈 상황
letsEatSomething() // 네가 배가 고픈 상황
letsEatSomething() // 내가 심심한 상황
```

- 다른 개발자를 배려하는 주석의 유형:
  - **이유를 알려주는 것**: 모바일에서 테스트했더니 0.5 이상은 품질 차이가 없었다. / 로딩 속도는 2배 빨랐다.
  - **개발자가 새롭게 발견한 것**: 놀랍게도 이 방식이 기존보다 결괏값을 2배 빠르게 보여준다. / 난 단지 순서만 바꿨을 뿐이다.
  - **예상 질문과 답**: 잘 보면 왜 마지막 대화 ID를 기록하는지 궁금할 것이다. / 팀 사이트에서 LastTalkId 변수를 검색해서 확인하자.

### 주석도 코드다

- 잘못된 주석이 늘어나면 개발자가 주석을 믿지 않는 풍토가 생기고, 그러면 더 주석에 신경 쓰지 않게 되는 **악순환의 늪**에 빠짐.
- **주석도 코드라고 생각**하고 코드 리뷰 시 주석 리뷰도 꼼꼼히 해야 함.
- 불필요한 주석은 없애고, 꼭 필요한 주석은 **반드시 코드처럼** 다뤄야 함.
