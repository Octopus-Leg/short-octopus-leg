# 3장 | 사용자와 소통하는 에러 메시지 쓰기

## 01 에러 메시지를 쓰기 전에 에러부터 없애자
### 404 에러가 죄송할 일인가?
- 사용자가 URL을 직접 잘못 입력하여 발생하는 **404 에러**는 개발자 과실이 아님.
- 반면 사이트 내 링크를 클릭했는데 해당 페이지가 없는 **깨진 링크**로 인한 404 에러는 개발자의 책임임.
### 깨진 링크는 개발자의 책임이다
- **구글 서치콘솔**과 같은 검색 엔진 도구를 활용하면 사이트 내 깨진 링크를 효율적으로 탐색 가능함.
### 개발자용 에러 메시지와 사용자용 에러 메시지를 분리하자
- **개발자용**과 **사용자용** 에러 메시지를 분리해서 사용자에게는 사용자용 에러 메시지만 보여주는 것이 좋음.
    ```c
    #define ERROR_MESSAGE_FOR_USER_911 "시스템 점검 중입니다."
    #define ERROR_MESSAGE_FOR_DEVELOPER_911 "Database Connection Timeout at Pool-1"
    ```

## 02 사용자 에러 메시지를 제대로 쓰는 법
### 사용자 에러에 대처하는 메시지
- 사용자가 오류를 스스로 해결할 수 있도록 다음의 세 가지 요소를 포함해야 함.
    1. **에러 내용**: 발생한 문제의 종류 (ex. 회원가입을 진행할 수 없습니다.)
    2. **에러 원인**: 오류를 발생시킨 근본적인 이유 (ex. 휴대전화 번호를 잘못 입력하셨습니다.)
    3. **해결 방법**: 사용자가 취해야 할 구체적인 행동 (ex. 휴대전화 번호 입력란에는 숫자만 입력하십시오.)
### 에러 메시지를 보여주는 순서
- 원인이 복잡한 경우 사용자의 편의를 위해 **에러 해결 방법**을 최상단에 배치하는 것이 효과적임.
    ```text
    3초 후에 다시 시도하십시오.
    (이하 생략된 에러 원인과 내용)
    ```
- 에러 내용과 원인을 합쳤을 때 문장이 매끄럽지 않다면 에러 내용을 과감히 생략하고 **원인과 해결 방법** 위주로 간결하게 작성함.
### 오락가락 메시지와 버튼 메시지
- 사용자의 오해를 방지하기 위해 모호한 메시지 대신 **행동**에 집중하여 기술해야 함.
- 버튼 텍스트에 단순한 `예/아니오`를 사용하는 것보다 버튼의 본래 역할은 특정한 행동을 유도하는 것이므로 `이 페이지에서 머물기/이 페이지에서 나가기`처럼 수행될 동작을 명확히 명시하는 것이 바람직함.
- 버튼 텍스트가 조금 길어지더라도 짧고 의미가 애매한 것보다는 길고 분명한 것이 나음.

  > **안 좋은 예시**
  > 
  > 지금 이 페이지를 떠나면 편집한 내용이 취소될 수 있습니다. 취소하시겠습니까? (예/아니오)

  > **좋은 예시**
  > 
  > 편집한 내용을 **삭제**하고 다른 페이지로 **이동**하시겠습니까? (이 페이지 머물기 / 이 페이지 나가기)


## 03 사용자의 에러를 줄이는 메시지 구조화
### 버튼의 순서
- 버튼에서 **확인-취소 순서**가 서비스나 운영체제마다 다르지만 OS를 따르기보다 서비스 내의 **일관성**을 유지하여 사용자 경험이 어긋나지 않도록 해야 함.
- 필요하다면 시각적 강조를 통해 `다른 사이트로 이동하기`보다 `이 사이트에 머물기` 버튼이 더 눈에 띄게 배치하고 특정 버튼을 미리 선택해 두는 것도 방법임.
### 사용자의 반복 에러를 막는 법
- 사용자가 기억을 잘 떠올려 최소한의 시도로 로그인을 하도록 **남은 로그인 횟수**를 메시지로 보여주면 자기 행동에 주의하게 됨.

## 04 에러 메시지 대신 예방 메시지를 쓰자
### 서비스를 이해하면 에러를 예방할 수 있다
- 호텔 예약 앱에서 사용자가 오늘 이전 날짜를 선택했을 때 에러 메시지를 출력하기보다 아예 **선택할 수 없게** 설계하는 것이 좋음.
### 사용자를 이해하면 에러를 예방할 수 있다
- 신용카드 16자리를 입력할 때 **4자리씩 4등분**으로 나누어 입력하게 구성하면 사용자의 오타 에러를 줄일 수 있음.
- **CAPS LOCK**이 켜져 있는 사실을 미리 알려주는 것만으로도 로그인 오류 예방이 가능함.
### 닭이 먼저? 알이 먼저?
- **재확인 방식**(요청 시 재확인)과 **취소 방식**(우선 요청 후 취소 기능 제공) 중 무엇을 택할지는 개발자의 철학 문제임.
- 에러 메시지를 보여주기 전에 스스로 사용자를 어떤 관점으로 보는지 생각해야 함.
- 사용자를 **불완전한 존재**로 인식하는 순간 모든 행동에 **경고**로 대응하게 되므로 주의가 필요함.
