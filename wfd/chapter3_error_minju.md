# 3장 사용자와 소통하는 에러 메시지 쓰기

## 01 에러 메시지를 쓰기 전에 에러부터 없애자

### 친절한 404, 불친절한 404

- 사용자가 의도 없이 개발자의 산출물을 직접 보게 되는 대표적인 경우가 **HTTP 404 에러 메시지**임.
- **404**: 클라이언트가 서버와 통신은 가능하지만, 서버가 요청한 페이지를 찾을 수 없음을 나타내는 HTTP 표준 응답 코드
- 구글의 404 페이지는 단순하고 불친절한 반면, 위키피디아·다음·YES24 등 국내 사이트는 고객센터 링크, 대안 URL 제안 등을 포함해 상대적으로 친절함.
  - 위키피디아: 페이지 미발견 안내 + 의도한 URL 추측 제안(`Did you mean: /wiki/404`) + 관련 링크 제공
  - 다음·YES24·네이버·넷마블·우리은행: 사과 문구 + 고객센터 링크 + 홈 이동 버튼 등 제공

### 404 에러가 죄송할 일인가?

- 사용자가 404 에러를 만나는 경우는 두 가지뿐임.
  1. 사용자가 URL을 잘못 입력한 경우 → 사용자의 문제이므로 개발자가 죄송할 이유는 없음.
  2. 사용자가 링크를 클릭했으나 해당 페이지가 없는 경우 → **개발자의 책임**임.
- 죄송해야 할 상황은 두 번째 경우뿐이며, 개발자는 깨진 링크를 수정해야 함.

### 깨진 링크는 개발자의 책임이다

- **깨진 링크(broken link / dead link / bad link)**: 사용자가 링크를 클릭했을 때 해당 페이지가 없는 경우
- 깨진 링크가 있다는 것은 개발자가 자기 소임을 다하지 못했다는 의미임.
- 사이트 내부 깨진 링크는 **브로큰링크체크닷컴(https://www.brokenlinkcheck.com)** 같은 서비스로 확인 가능함.
- 외부 사이트에서 연결된 깨진 링크는 **구글 서치콘솔(https://www.google.com/webmasters/tools/)** 을 통해 확인하고 수정할 수 있음.

### 개발자용 에러 메시지와 사용자용 에러 메시지를 분리하자

- 개발 중 작성한 에러 메시지(예: `code:911\n이거 나오면 야근.ㅠㅠ`)가 사용자에게 노출되는 사고를 예방해야 함.
- C언어의 `#define` 전처리기처럼 **처음부터 개발자용과 사용자용 메시지를 분리**해 작성하는 것이 좋음.
  ```c
  #define ERROR_MESSAGE_FOR_USER_911       "현재 개발 중인 기능이어서 에러가 생길 수 있습니다. 양해 부탁합니다."
  #define ERROR_MESSAGE_FOR_DEVELOPER_911  "code:911\n이거 나오면 야근.ㅠㅠ"
  ```

## 02 사용자 에러 메시지를 제대로 쓰는 법

### 사용자 에러에 대처하는 메시지

- 에러 메시지의 목적은 에러가 났음을 알리는 것이 아니라, **사용자 스스로 에러를 해결하게 하는 것**임.
- 따라서 사용자 에러 메시지에는 세 가지가 반드시 포함되어야 함.
  - **에러 내용**: 오류로 인한 문제와 종류
  - **에러의 원인**: 오류를 발생시킨 직접적이고 근본적인 원인
  - **에러 해결 방법**: 사용자가 오류를 해결할 가장 쉽고 빠른 방법

> **나쁜 예**: "회원가입을 진행할 수 없습니다."
> **좋은 예**: "휴대폰 번호를 잘못 입력하셔서 회원가입을 진행할 수 없습니다. 휴대폰 번호 입력란에는 숫자만 입력하십시오."

### 에러 메시지를 보여주는 순서

- 에러 내용·원인이 간단한 경우: **내용 → 원인 → 해결 방법** 순으로 표시함.
- 에러 내용·원인이 복잡한 경우: **해결 방법을 먼저** 제시하는 것이 사용자에게 더 효과적임.
  1. [에러 해결 방법] 3초 후에 다시 시도하십시오.
  2. [에러 원인] 아이템을 인계받을 상대방에게 다른 사용자가 아이템을 인계하는 중이어서 동시에 인계할 수 없습니다.
  3. [에러 내용] 요청하신 아이템의 인계를 시간 내에 처리하지 못했습니다.
- 에러 내용과 원인을 합칠 경우 문장이 복잡해지면, **에러 내용을 생략하고 원인만 간단히** 써도 해결 방법이 먼저 나오므로 사용자가 충분히 이해할 수 있음.

### 오락가락 메시지와 버튼 메시지

- 알림창에 버튼이 두 개일 때, **'예/아니오' 대신 수행할 행동을 직접 표시**하는 것이 좋음.
  - 나쁜 예: "지금 이 페이지를 떠나면 편집한 내용이 취소될 수 있습니다. 취소하시겠습니까?" + [예] [아니오]
  - 좋은 예: "편집한 내용을 삭제하고 다른 페이지로 이동하시겠습니까?" + [삭제하고 이동하기] [계속 편집하기]
- '취소'라는 단어를 두 번 쓰면 상황이 모호해지므로, **행동에만 집중한 버튼 텍스트**를 사용해야 함.

## 03 사용자의 에러를 줄이는 메시지 구조화

### 버튼의 순서

- 페이스북: **취소(머물기) → 확인(나가기)** 순서 (방금 한 행동을 취소하는 것이 먼저).
- 크롬 브라우저: **확인(나가기) → 취소** 순서 (확인–취소 문제).
- Windows: 확인이 먼저, macOS·Android·iOS: 확인이 오른쪽.
- 확인이 오른쪽에 있는 이유는 글쓰기·방향 표시가 왼쪽→오른쪽이라 **행동의 연속성** 때문임.
- 국제 표준이 없으므로, OS보다는 **서비스 내에서 일관성**을 유지하는 것이 중요함.
- 필요하다면 회사가 유도하고 싶은 방향의 버튼을 **시각적으로 강조**하는 것도 좋은 방법임.

### 사용자의 반복 에러를 막는 법

- 같은 에러를 반복할 때는 일정 횟수 이후 **다른 방법이나 다른 메시지**를 보여줘야 함.
- 네이버 로그인 실패 예시: 4회까지는 동일한 에러 메시지 → 5회째부터는 자동입력 방지 문자 요구.
  - 문제점: 갑작스러운 메시지 변화로 사용자가 당황함.
  - 개선책: **남은 로그인 횟수를 메시지로 표시**하여 사용자가 자기 행동에 주의하게 만듦.
- 인터넷 뱅킹 예시: 비밀번호 오류 시 남은 횟수를 미리 알려줌으로써 사용자 불편과 비용을 줄임.

## 04 에러 메시지 대신 예방 메시지를 쓰자

### 서비스를 이해하면 에러를 예방할 수 있다

- 에러를 줄이려면 사용자의 사용 방식을 충분히 이해하고 분석해야 함.
- 호텔 예약 달력 예시: 과거 날짜를 선택하면 에러가 발생하므로, **처음부터 과거 날짜를 비활성화**하면 에러 자체가 사라짐.
  - 체크인 날짜 선택 후 → 체크아웃은 체크인 이후 날짜만 선택 가능하도록 제한.

### 사용자를 이해하면 에러를 예방할 수 있다

- 긴 숫자 입력 시 실수를 줄이기 위해 **4자리씩 구분된 입력 폼**을 제공함(신용카드 번호 등).
- 이메일 입력 시 **@ 뒷부분을 드롭다운으로 선택**할 수 있게 해 오타를 방지함.
- 비밀번호 입력란에서 **Caps Lock 활성화 여부를 미리 알려줌**으로써 로그인 실패를 예방함.
- 배터리 부족 시 갑자기 기기가 꺼지지 않도록 **예방 메시지를 미리 보내** 사용자가 스스로 결정하게 함.
- 서비스와 사용자를 이해하면 에러 메시지 대신 **예방 메시지**를 제공할 수 있음.

### 닭이 먼저? 알이 먼저?

- 결재 요청 시 확인 방식과 취소 방식 비교
  - **재확인 방식**: 결재 요청 → 재확인 → 결재 처리
  - **취소 방식**: 결재 요청 → 결재 처리 + 취소 기능
  - **혼합 방식**: 결재 요청 → 재확인 → 결재 처리 + 취소 기능
- 재확인 방식은 구현이 쉽지만, **사용자를 불완전한 존재로 보는 철학**에 기반함.
- 취소 방식은 구현이 번거롭지만, 사용자가 자신의 행동을 **주체적으로 책임**지게 만듦.
- 어떤 방식을 쓸지는 서비스·사용자에 따라 다르며, **개발자 스스로의 철학**이 필요함.
- 에러 메시지를 쓰기 전에 그 메시지가 꼭 필요한지, 본래 역할을 제대로 수행하는지 고려해야 함.
