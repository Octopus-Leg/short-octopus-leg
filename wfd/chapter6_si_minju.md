# 6장 수주를 돕는 SI 제안서 쓰기

## 01 개발자가 알아야 할 제안서 작성 원칙

### 개발자와 제안 PM의 차이

- 개발자는 제안서에서 주로 **기술 부문**을 담당하며, 제안 PM이 비개발자 출신이면 기술 부문 전체를 개발자가 책임져야 함.
- 기술 부문은 대부분 그림·표로 구성되지만, 개발자가 쓰기 가장 어려운 것은 **전략적 서술**임.

```
제안 PM이 개발자에게 요구하는 것
  - "기술 부문의 전략을 한마디로 말해 주세요."
  - "우리가 제안하는 시스템 구성도의 차별점을 두 줄로 써 주세요."
  - "심사위원에게 던지는 메시지를 한 문장으로 써 주세요."
```

### 시스템 구성도의 본질은 그림이 아니다

- 시스템 구성도(As-Is → To-Be)를 그리는 것은 개발자의 본래 업무이지만, **목적·목표·전략·방안·효과를 기술하는 것은 그 이상**임.
- `"시스템의 성능, 안정성, 확장성을 위해 다음과 같이 시스템을 구성함"` → 어떤 구성도에도 쓸 수 있는 표현이므로 의미 없음.
- **'시스템 구성도'** 항목에는 아래 요소가 모두 포함되어야 논리적으로 납득 가능함.

| 포함 요소 |
|:---|
| 고객의 요구 / 고려사항 |
| 구성 전략 / 구성 목표 / 구성 방안 |
| 구성의 특장점 / 기대 효과 |

### 첫째, 제안요청서 분석

- **제안요청서(RFP)** 에는 제안의 문제와 답이 모두 담겨 있음.
- 많은 개발자가 기술·요구사항 부분만 읽고 제안서를 작성하는 경향이 있으나, RFP 곳곳에 기술 부문 작성 힌트가 숨어 있음.
- RFP에 포함된 항목: 제안 요청 배경, 사업 목적, 요구 사항, 목표 시스템, 현재 시스템 등.

### 둘째, 논리적 완결성

- 제안서는 소설이 아니므로, 심사위원은 관심 있는 항목만 골라 읽음.
- **각 항목은 독립적으로 완결**되어야 하며, 앞 내용을 참조하지 않아도 이해 가능해야 함.
- 항목 간 내용이 순서 의존적으로 연결되면 제안서는 '숨은그림찾기'가 되어 심사위원의 마음을 잃게 됨.

## 02 고객의 문제 인식과 제안사의 문제 해결 능력

### 문제 인식과 문제 해결 능력

- 제안서의 시작은 문제가 아니라 **고객의 문제 인식**임.
- **고객의 문제 인식 수준**과 **제안사의 문제 해결 능력**, 두 축의 조합으로 4가지 제안 전략이 도출됨.

|  | 제안사 능력: **탁월** | 제안사 능력: **부족** |
|:---:|:---:|:---:|
| 고객 인식: **중대** | ① 경쟁사와 비교하여 제안하라 | ② 일단 동감하고 다른 방안을 제시하라 |
| 고객 인식: **사소** | ③ 고객이 문제를 중대하게 인식하게 만들어라 | ④ 경쟁사의 전략을 확인해서 대처하라 |

### ① 경쟁사와 비교하여 제안하라

- 제안사의 강점을 최대한 부각하는 방법은 경쟁사와 **세부 기능·스펙을 비교**하는 것임.
- 솔루션이 **혁신적이면서도 안정적**이라는 점을 동시에 강조해야 함. 혁신성만 강조하면 고객이 너무 실험적이라고 오해할 수 있기 때문임.

### ② 일단 동감하고 다른 방안을 제시하라

- 경쟁사와 정면으로 부딪혀서는 이길 승산이 없으므로, **무조건 다른 접근법**을 찾아야 함.
- 제안에서는 누가 더 일리 있는 안을 내놓느냐가 중요함.

```
전략 순서
  1. 고객의 문제 인식에 동감
  2. 경쟁사 방식의 문제점을 논리적으로 제시
  3. 자사의 대안이 더 낫다는 것을 설명
```

### ③ 고객이 문제를 중대하게 인식하게 만들어라

- 고객에게 기존 방식의 문제를 부각하고, 선진국·선진사의 트렌드와 사용자 니즈를 알려줌으로써 문제 인식을 높여야 함.
- 여러 방안을 비교표로 제시하고, 제안사의 방안이 더 낫다는 것을 논리적으로 설명함.

### ④ 경쟁사의 전략을 확인해서 대처하라

- 경쟁사가 어떻게 나올지 미리 알 수 없더라도, 사소한 문제에 대한 **예상 질문과 답 정도는 준비**해야 함.
- 아무리 고객이 사소하게 여기고 제안사도 딱히 내세울 것이 없다 해도 **할 말은 준비**해야 함.

## 03 고객의 요구사항은 변할 수밖에 없다

### 개발은 고객 요구 실현

- 요구사항은 개발 중에 바뀌고, 새로 추가되고, 완성한 기능이 빠지기도 함.
- **최초의 요구사항은 단지 참고사항일 뿐**이며, 요구사항이 끊임없이 변한다는 것을 이해해야 대응할 수 있음.

### 요구사항을 분석하지 말고 제시하라

- 고객은 자기가 원하는 제품이 정확히 뭔지 모름.
- 개발자는 고객의 요구사항을 받아서 분석하는 것이 아니라, **고객에게 요구사항을 제시해서 고객이 선택하게** 만들어야 함.
- 상충하는 요구사항은 현실에서 구현 불가능하므로, 개발자가 문제를 명확히 알려주고 **대안을 제시**해야 함.

```
상충 요구사항 예시
  - 반응형 웹 구축
  - 한 화면의 내용 길이를 최대 3페이지로 할 것
  - 1024×768 해상도에 맞출 것
  → 반응형 웹은 단말기·브라우저 크기에 반응하므로 1024×768 고정 불가
```

### 변화하는 요구사항에 대비하라

- 기존 방식은 요구 정의와 고객 검수 사이의 **시간 차가 길어** 변경에 취약함.
- 이를 **투 트랙(Two-Track) 방식**으로 전환함.

| 트랙 | 범위 | 요구 수준 | 목적 |
|:---:|:---:|:---:|:---:|
| 1트랙 | 목표 시스템 전체 | 기본·일반적 수준 | 전체 구조 정의 |
| 2트랙 | 기능별 | 세부·정밀 수준 | 개발 직전 재정의 + 즉시 검수 |

- 투 트랙 방식을 통해 요구 정의와 검수 사이의 **시간 차를 최소화**하고 고객의 변덕에 대비할 수 있음.

## 04 고객의 총 만족도를 높이자

### 요구라고 다 같은 요구가 아니다

- 개발자가 고객의 요구를 충족했다고 해서 고객이 무조건 만족하는 것은 아님.
- 요구사항을 **모두 충족**하는 개념이 아니라, **고객의 총 만족도를 높이는** 방향으로 개발자의 자원을 설계해야 함.
- 요구사항별로 개발자의 소요 시간과 고객의 예상 만족도를 수치화하여 XY 좌표로 표시하면 개발 우선순위를 파악할 수 있음.

| 요구사항 | 개발자 소요 시간(h) | 고객 예상 만족도(1~10) |
|:---:|:---:|:---:|
| A 기능 | 20 | 3 |
| B 기능 | 60 | 1 |
| C 기능 | 10 | 9 |

> C 기능을 가장 먼저 만들고 시간을 조금 더 들여서 테스트해야 함.

### 카노 모델로 본 요구의 세 가지 종류

- **카노 모델(Kano Model)**: 카노 노리아가 1980년대에 연구한 상품 기획 이론. 기능·구성 요소의 충족에 대한 소비자 만족의 관계를 설명함.
- x축: 고객의 요구 충족도 / y축: 고객의 만족도

| 유형 | 특징 | 예시 | 개발자 전략 |
|:---:|:---:|:---:|:---:|
| **① 기본 기능** | 없으면 불만족, 있어도 만족도 크게 오르지 않음 | 회원가입, 로그인 | 모두 수용 |
| **② 기능의 성능** | 충족할수록 만족, 미충족 시 불만족 | 페이지 로딩 시간 | 한계를 정함 |
| **③ 특별한 기능** | 없어도 불만 없지만, 있으면 크게 만족 | 지문·홍채 인식 로그인 | 하나쯤 만들어야 함 |

- 만족도는 시간이 지나면서 달라짐. **지금의 특별 기능도 나중에는 기본 기능**이 될 수 있음.
- 제안요청서·계약서에 없더라도 특별한 기능 하나가 개발자의 미해결 요구나 버그 문제를 해결해 줄 수도 있음.
