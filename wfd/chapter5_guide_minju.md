# 5장 설명, 묘사, 논증, 서사로 개발 가이드 쓰기

## 01 서비스 개념을 범주, 용도, 특징으로 설명하자

### 범주, 용도, 특징

- 서비스 매뉴얼이나 개발 가이드의 첫 문단은 서비스 개념을 설명하는 자리. 독자가 첫 문단에서 서비스 개념을 확실히 잡지 못하면 이후에 나오는 내용을 제대로 이해할 수 없음.
- 서비스 개념을 설명할 때는 **범주 → 용도 → 특징** 순으로 쓰는 것이 좋음.
  - **범주**: 독자가 이미 아는 것을 먼저 가져다가 찬찬히 설명해 나가는 방법. 독자가 안심하며 머릿속으로 해당 범주를 떠올림.
  - **용도**: 독자가 자연스럽게 서비스의 용도와 같음을 알아차림.
  - **특징**: 그 범주 내의 유사 서비스와 차별화하는 특징을 말하면 독자는 서비스의 개념을 정확히 이해함.
- 예: AWS의 Amazon S3 첫 문단

| 구분 | 내용 |
|---|---|
| 범주 | Amazon S3는 인터넷 스토리지 서비스입니다. |
| 용도 | Amazon S3를 사용하면 인터넷을 통해 언제 어디서든 원하는 양의 데이터를 저장하고 검색할 수 있습니다. |
| 특징 | AWS Management Console의 간단하고 직관적인 웹 인터페이스를 통해 이러한 작업을 수행할 수 있습니다. |

### 범주를 정확하고 적절하게 선택하자

- 범주는 서비스를 소개하거나 설명하는 첫 문장인 만큼 **정확하고 적절하게** 정해야 함.
- 가장 좋은 방법은 여러 경쟁사가 사용하는 **보편적인 서비스 범주를 따라** 하는 것. 경쟁사 홈페이지의 설명을 보면 적절한 범주를 파악할 수 있음.
- 범주는 서비스의 정체성을 나타낼 뿐만 아니라 **검색어와도 연관**이 있어 신중하게 정해야 함.
- 강력한 마케팅을 전개할 수 있다면 완전히 새로운 개념의 범주를 만들어 사용할 수 있음. 범주를 선점하는 회사가 해당 범주의 대표주자가 되기 때문.
- ABC가 크라우드펀딩 범주에 속하지만 차이가 있다면 수식을 한정해 차이를 돋보이게 할 수 있음.

| 상황 | 예시 |
|---|---|
| 경쟁사보다 전반적으로 발전한 서비스 | ABC는 보다 발전된 크라우드펀딩 서비스입니다. |
| 특정 사용자를 위한 서비스 | ABC는 개발자를 위한 크라우드펀딩 서비스입니다. |
| 새로운 범주를 만들고 싶을 때 | ABC는 크라우드옥션 서비스입니다. |

### 용도를 범주의 핵심 기능으로 기술하자

- 첫 문단의 두 번째 문장은 **독자 관점의 용도**를 주로 적음. 용도는 독자가 이 서비스를 이용하는 이유.
- 독자가 서비스 매뉴얼이나 개발 가이드를 읽는 이유는 서비스의 **핵심 기능**을 사용하기 위해서. 따라서 서비스의 핵심 기능을 쓰면 용도를 기술할 수 있음.
- 서비스의 핵심 기능은 서비스가 속한 **범주의 핵심 기능과 같음**.
- 여러 핵심 기능들을 추상적인 표현으로 연결하여 한 문장으로 만들면 용도가 됨.

| 구분 | 내용 |
|---|---|
| 개선 전 | 인공지능 플랫폼을 사용하면 사용자의 앱/스피커에서 음성과 이미지를 인식하여 인공지능 대화, 자동 추천, 신경망 기반 자동번역 등을 통해 사용자의 콘텐츠/서비스에 연결할 수 있습니다. |
| 개선 후 | 인공지능 플랫폼은 사용자의 요구를 인식하여 인공지능으로 분석해서 사용자가 원하는 정보나 서비스를 제공합니다. |

### 특징을 장점과 강점에서 뽑아 쓰자

- 범주와 용도에는 보편적인 내용을 적음. 하지만 **특징은 차별화하는 내용**을 적어야 함.
- **장점**: 자기 기준에서 잘하는 것. **강점**: 경쟁 서비스와 비교해서 나은 것.
- 서비스 개념에서 특징을 쓸 때는 **장점이자 강점인 것**을 쓰는 것이 가장 좋음. 장점이자 강점인 것이 없다면 장점과 강점을 합쳐서 한 문장으로 쓰는 것도 좋은 방법.
- 경쟁사와 기능별로 5점 척도로 점수를 매기면 장점과 강점을 동시에 비교할 수 있음.

| 구분 | 예시 |
|---|---|
| 장점이자 강점을 하나로 쓴 경우 | ABC 서비스는 최고의 안정성을 제공합니다. |
| 장점과 강점을 한 문장에 합친 경우 | ABC 서비스는 간단하고 직관적인 웹 인터페이스와 안정적인 서비스를 제공합니다. |

## 02 정확히 이해하도록 그림과 글로 묘사하자

### 글에 묘사를 더하면 이해가 빠르다

- 묘사는 어떤 대상이나 사물, 현상을 언어로 서술하거나 그림을 그려서 표현하는 것.
- 언어로만 묘사하려면 작가와 독자가 어느 정도 같은 경험을 가져야 함. 개발문서가 특히 그러함.
- ABC 서비스를 개발해본 적이 없는 개발자에게 개발 방법을 알려주는 문서는 언어로만 묘사하기에 한계가 있어 **그림을 그려 넣어야 할 때가 많음**.
- 이메일 제목 하나에도 묘사를 더하면 이해가 빨라짐.

| 묘사 유형 | 예시 |
|---|---|
| 시각 묘사 | 상다리가 부러질 뻔한 워크숍 사진 공유 |
| 청각 묘사 | 소곤소곤 깊은 대화를 나눈 워크숍 사진 공유 |
| 후각 묘사 | 코끝을 톡 쏘는 화이트와인과 함께한 워크숍 사진 공유 |
| 미각 묘사 | 씹고! 물고! 뜯고! 워크숍 사진 공유 |
| 촉각 묘사 | 온몸으로 부대끼며 열정을 태운 워크숍 사진 공유 |

### 글과 그림의 내용을 일치시키자

- 개발문서에는 글과 코드만 들어가지 않음. 그림이나 사진이 많음.
- **글이 그림과 같이 있으면 글과 그림이 같은 용어를 사용하는지 꼭 확인**해야 함. 같은 것을 다르게 표현하는 순간 독자는 헷갈리기 시작하고 그 결과는 에러로 나타남.
- 글을 먼저 쓰고 그림을 나중에 그리든, 그림을 먼저 쓰고 글을 나중에 쓰든 상관없음. 중요한 것은 **같은 것을 같은 용어로 표현**하는 것.
- 예: Clova Chatbot 작동 구조 문서에서 그림의 'boundary term'을 글에서 '경계 문구'로 달리 표현하면 독자는 둘이 같은 것인지 금방 찾을 수 없음.

| 구분 | 예시 |
|---|---|
| 잘못된 표기 | 경계 문구 |
| 올바른 표기 | boundary term, 바운더리 텀, 경계 문구(boundary term) |

### 객관적 묘사와 주관적 묘사 둘 다 하자

- 묘사에는 **주관적 묘사**와 **객관적 묘사**가 있음.

| 묘사 유형 | 예시 |
|---|---|
| 주관적 묘사 | "저기 앞에 큰 사거리에서 오른쪽으로 돌아서 5분쯤 가면 기차역이 보여요." |
| 객관적 묘사 | "두 번째 사거리에서 10시 방향으로 돌아서 300미터 가면 우측에 기차역이 있어요." |

- 개발 과정에서 나오는 문서는 처음에는 주관적 묘사가 많다가 객관적 묘사가 크게 늘고, 마지막에는 주관적 묘사가 다시 늘어나는 경향이 있음.
- **요구사항 정의서**는 주관적 묘사를 완전히 배제하고 오직 객관적 묘사만 포함해야 함.
- **FAQ 답변**처럼 최종 사용자에게 전달하는 문서는 주관적 묘사로 바꿔 설명하면 훨씬 이해하기 쉬움.
- 개발자가 주관적 묘사와 객관적 묘사 둘 다 잘하면 기획자나 디자이너와 협의할 때 **주도권**을 가질 수 있음.

## 03 논증으로 유용한 정보를 제공하자

### 의견을 쓰려면 근거를 대자

- 논증은 옳고 그름을 이유나 근거를 들어서 밝히는 것. 논증은 객관적이고 논리적인 방식으로 어떤 사실을 증명해서 상대를 설득하는 일.
- 개발문서에도 꽤 많은 논증이 있음. SDK 설정 값 설명 문서가 대표적인 예.
- **의견만 말하고 근거를 말하지 않으면** 독자가 직접 테스트해볼 수밖에 없음.
- 가장 좋은 방법은 **개발자가 직접 체험한 결과를 알려주는 것**.

| 구분 | 예시 |
|---|---|
| 의견만 | default 값은 10이며, 30가량의 값을 사용해도 괜찮습니다. |
| 근거 포함 | default 값은 10이며, 30까지를 권합니다. (테스트 결과, 30까지는 인코딩 시간이 10~20%로 소폭 늘었습니다. 35일 때는 100%, 40일 때는 200%로 인코딩 시간이 급증했습니다.) |

### 거칠게도 공손하게도 쓰지 말자

- 개발문서에서 모호한 표현은 독자에게 혼란을 줌. 개발자가 독자에게 여지를 10% 주면 독자는 30%, 50%로 부풀림.
- 거칠거나 공손한 표현 대신 **명확한 지시형**으로 통일함.

| 구분 | 예시 |
|---|---|
| 거친 표현 | 캔버스 뷰 없이 합성을 시도할 수 있습니다. 그러나 SDK에 대한 깊은 이해를 요하며 캔버스 뷰를 사용하는 합성보다 매우 어렵습니다. |
| 공손한 표현 | ~해야 합니다만 반드시 해야 하는 것은 아닙니다. / ~하면 좋습니다. 다만 ~한 경우에는 안 해도 무방합니다. |
| 올바른 표현 | ~하십시오. / ~하지 마십시오. |

### 주장과 이유의 거리를 좁혀서 쓰자

- 주장을 했으면 **이유를 바로 이어서** 말해야 함.
- 개발문서는 처음부터 순서대로 읽지 않음. 그때그때 필요한 부분만 찾아보기 때문에 **주장을 말했으면 중복되더라도 이유를 함께 말하거나 이유를 찾을 수 있는 곳을 알려줘야** 함.

| 구분 | 예시 |
|---|---|
| 주장만 | 마지막 대화 ID를 기억·갱신하지 않으면 치명적인 문제가 생깁니다. |
| 주장 + 이유 링크 | 마지막 대화 ID를 기억·갱신하지 않으면 CIC가 요청을 처리하는 동안 사용자의 요청 변경에 대응할 수 없는 치명적인 문제가 생깁니다. 자세한 내용은 CIC > 개요 > 간접 대화 구조에서 확인하십시오. |

### 문제와 답의 거리를 좁혀서 쓰자

- 개발문서가 사실상 문제 해결 문서이므로 **문제 바로 다음에 답이 먼저 나와야 함**. 답을 찾아가는 지루한 과정을 먼저 보여줄 필요가 없음.
- 이유를 설명하기 너무 길다면 **일단 짧게라도 설명하고 이유가 있는 곳을 알려주는 방법**을 사용함.

| 구분 | 예시 |
|---|---|
| 답이 마지막 | 주소록 목록 > 주소록 대시보드 > 그룹 목록 > 구독자 목록으로 이동하면 브라우저에 표시되는 URL에서 'subscribed' 뒤의 숫자가 groupid입니다. |
| 답이 먼저 | 구독자 목록과 관련한 groupid를 확인하려면 '구독자 목록' 페이지 URL에서 'subscribed' 뒤의 숫자를 보십시오. |

## 04 서사를 활용해 목차를 만들자

### 개발과 서사

- 서사는 사실을 있는 그대로 순서대로 적는 것. 누가 어떤 행동을 했고, 그 행동으로 어떤 일이 일어났는지 그대로 쓰는 것.
- 개발에서도 서사를 많이 씀. 소설과 다른 점은 독자에게 일어난 사건을 알려주는 것이 아니라 **어떤 일을 하라고 지시**한다는 것. 스크린샷을 보여주면서 순서대로 말할 때가 많음.
- 이때는 글이 주가 아니라 **스크린샷이 주**가 됨. 그림이나 사진이 글과 붙어 있는 잡지처럼 글을 써야 함.
- 번호별로 글이 많지 않은 경우에는 **한 문장으로 요약하고 단어 앞에 번호를 붙이는 것**이 좋음.

| 구분 | 예시 |
|---|---|
| 번호별 문장 방식 | 4. APNS 섹션에서 첫 번째 칸에 '인증서 업로드'를 클릭하여 인증서를 등록합니다. |
| 한 문장 요약 방식 | 푸시 서비스를 설정하려면 ①설정 > ②푸시 화면에서 ③푸시 사용을 켜고, ④인증서를 업로드한 뒤 ⑤비밀번호를 설정하여 ⑥저장합니다. |

### 독자의 수준 대신 기술의 범용성을 기준으로 쓰자

- 개발문서에서 서사는 결국 **개발자와 독자 사이의 줄다리기** 같은 것. 개발자는 의미 없는 것은 빼고 중요한 것은 넣기를 원하고, 독자도 의미 있는 것만 남고 무엇이 중요한지 구별해 주기를 원함.
- 독자의 수준을 기준으로 하면 서사가 너무 길어지거나 짧아짐 → **기술의 범용성**을 기준으로 하는 것이 좋음.

| 기술 범용성 | 서술 방식 | 예시 |
|---|---|---|
| 범용적 | 짧게 씀 | 툴바에서 더 보기(:) > 확장을 클릭합니다. |
| 비범용적 | 자세하게 씀 | 컴퓨터에서 인터넷 브라우저를 엽니다. 인터넷 브라우저가 다 열리면 오른쪽 위에서 더 보기(:)를 찾습니다. ... |

- 개발문서를 읽기 전에 기본적인 수준을 맞춰놓는 방법도 있음: **'시작하기 전에'나 '개요'** 항목에 최소 분량으로 기본적인 지식을 설명함.

### 순서에서 단계를, 단계에서 목차를 만들자

- 지시 사항을 모두 순서대로 나열하면 글이 너무 길어짐. 이런 문제를 막기 위해 **단계 개념**을 사용함.
- 지시를 몇 개씩 묶어서 몇 가지 단계로 나눔. **각 단계는 반드시 목표**가 있어야 함. 이런 부류의 문서는 대부분 처음에는 쉽고 뒤로 갈수록 어려워짐. 그래서 첫 단계에서 쉬운 목표를 달성하면서 조금씩 어려운 단계의 목표를 달성하게 만들어야 함.
- 단계를 잘 만들었다면 **목차로 바로 쓸 수 있음**.

```
[Chrome DevTools에서 자바스크립트를 디버깅하기]

1단계: 버그 재현
  버그를 재현할 수 있어야 디버깅을 할 수 있습니다.
  1. Open Demo를 클릭합니다. ...

2단계: 실행 코드 검색 및 일시 중지
  DevTools에서는 코드를 실행하는 도중에 일시 중지하고 특정 시점의 모든 변숫값을 검사할 수 있습니다.
  1. 데모에서 Command+Option+I(Mac) 또는 Control+Shift+I(Windows, Linux)를 눌러서 DevTools를 엽니다. ...
```

- HTML로 개발문서를 게시할 때는 **사이드바에 목차를 배치해서 링크를 걸어주면** 독자가 보기 편함.
