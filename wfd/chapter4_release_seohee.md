# 4장 | 독자 관점에서 릴리스 문서와 장애 보고서 쓰기

## 01 체인지 로그를 분류, 요약, 종합하는 법
- 체인지 로그를 간단히 쓰면 한 일이 없어 보이고 자세히 쓰면 아무도 읽지 않아 쓸모가 없음. 적절한 양으로 쓰는 것이 중요함.
- 우선 체인지 로그를 최대한 많이 쓰고 나서 그것을 선정, 분류, 요약, 종합하는 기술이 필요함.
- **1단계: 선정하기**
  - 다음 표처럼 8가지 경우로 분류해 체인지 로그를 선정하고 순위나 수준을 결정할 수 있음.

    | 회사가 말하고 싶다 | 개발자가 말하고 싶다 | 독자가 듣고 싶다 | 순위 | 수준 | 예시 |
    | :---: | :---: | :---: | :---: | :---: | :--- |
    | O | O | O | 1순위 | 자세히 | 인공지능 추천 추가 |
    | O | O | X | 2순위 | 간단히 | 대용량 네트워크 증설 |
    | O | X | O | 3순위 | 간단히 | UI 개선 |
    | O | X | X | 4순위 | 주석 | 면책 조항, 정책 변경 |
    | X | O | O | 5순위 | 참고 | 작업 과정, 레퍼런스 |
    | X | O | X | 삭제 | - | 성과가 낮은 오류 수정 |
    | X | X | O | 삭제 | - | 경쟁력 없는 기능 추가 |
    | X | X | X | 삭제 | - | 사소한 오류 수정 |
- **2단계: 분류하기**
  - 비슷한 체인지 로그끼리 묶어야 함.
  - 개발 관점(독자가 개발자인 경우에 유용함.) ➡️ 새로운 기능 추가, 기능 개선, 오류 수정
  - 사용자 관점(독자가 일반 사용자인 경우에 유용함.) ➡️ 각 단계에 따라(게임 준비, 게임 중, 게임 종료)
- **3단계: 요약하기**
  - 서술식 문장을 개조식 문장으로 바꾸면 내용이 자연스럽게 축약되고 의미가 분명해짐.
  - ex) `미리 보기에서 간혹 리부팅되는 문제를 해결했습니다.` ➡️ `미리 보기 리부팅 문제 해결`
- **4단계: 종합하기**
  - 릴리스 내용 전체를 종합해서 한 문장으로 만들고 그것을 체인지 로그 첫 줄에 적어야 함.
  - 내용이 두 가지 이상이면 줄을 나누는 것이 좋음.
  - ex) `게임방에 더 빨리 입장 / 게임 결과 바로 확인`
- **전체 예시**
    ```markdown
    [사용자 편리성 개선]
    
    * 게임방에 더 빨리 입장
    * 게임 결과 바로 확인
    
    <세부 내용>
    * 게임 준비
      - 미리 보기 리부팅 문제 해결
      - 빈 게임방 자동 검색 기능 추가
      - 닉네임 만들 때 특수 문자 입력 기능 추가
    
    * 게임 중
      - 고해상도 폰에서 아이콘이 찌그러지는 오류 수정
      - 가로/세로 화면 전환 시 하단 메뉴가 사라지는 오류 수정
      - 애니메이션 스티커가 멈추는 오류 수정
    
    * 게임 종료
      - 최근 기록이 상위에 올라오도록 개선
      - 게임 종료 후 바로 순위를 볼 수 있도록 개선
      - 용량이 큰 사진을 등록할 때 휴대전화 메모리 사용 최소화
    ```
  
## 02 고객에게 유용한 정보를 쓰자
- 고객 관점으로 체인지 로그를 쓰려면 고객에게 끼치는 영향을 체인지 로그에 추가해서 기술하면 됨.
  - ex) `댓글에서 애니메이션 스티커 때문에 화면이 멈추는 문제를 해결했습니다. 이제 애니메이션 스티커를 정상적으로 댓글에 사용할 수 있습니다.(고객에 미치는 영향)`
- 고객의 문제 해결을 앞쪽으로 옮기고, 개발자의 문제를 짧게 줄여 개조식으로 바꾸는 것이 좋음.
  - ex) `애니메이션 스티커를 댓글에 정상 사용 가능(화면 멈춤 문제 해결)`
- 체인지 로그로 사용법을 유도할 수도 있고 하나의 관점이 아닌 내용에 따라 여러 관점으로 쓸 수 있음.
- **⭐️버전 가이드⭐️**
  > - 버전은 X.Y.Z 형태이며 각 숫자는 독립적인 자연수로 증가함. 점(.)은 소수점이 아닌 구분 기호이며 1.1.12는 1.1.9보다 높은 버전임.
  > - X는 Major, Y는 Minor, Z는 Patch로 이해하며 0으로 시작하는 버전은 초기 내부 개발용으로만 사용함. 최초 공개 API는 1.0.0이어야 함.
  > - X는 기존 버전과 호환되지 않는 변화가 있을 때 1씩 올림.
  > - Y는 기존 버전과 호환되는 새로운 기능이 추가됐을 때 1씩 올림. 큰 규모의 패치가 있을 때 Y를 올릴 수도 있으며, X가 올라가면 Y는 0으로 초기화함.
  > - 최초의 내부 개발 버전은 0.0.0이 아니라 0.1.0으로 시작함. 개발의 시작은 새로운 기능의 시작이므로 Y가 1이어야 하기 때문임.
  > - Z는 기존 버전과 완전히 호환되면서 작은 규모의 패치가 있을 때 1씩 올림. X나 Y를 올리면 Z는 0으로 초기화함.
  > - 정식 배포 전 사전 배포가 필요할 때는 Z 다음에 붙임표(-)로 구분하고 적절한 식별자를 적음. 알파 버전이 여러 개라면 1.0.0-alpha.1처럼 점(.)으로 구분하고 숫자를 덧붙임.
  > - 일단 배포된 버전은 변경해서는 안 되며, 변경 사항이 있다면 무조건 버전을 새로 올려야 함.

## 03 릴리스 문서는 문제 해결 보고서처럼 쓰자
- 문제에는 발생형, 탐색형, 설정형 세 가지 종류가 있음.
- **발생형 문제**
  - 우리 눈앞에 발생해 당장 걱정하고 해결하기 위해 고민하는 문제임.
  - 프로그램 에러가 대부분 발생형 문제임.
- **탐색형 문제**
  - 현재 상황을 개선하거나 효율을 높이는 문제임.
  - 눈에 잘 보이지 않아 그냥 놓아두면 큰 손실이 따르거나 해결할 수 없는 문제가 될 수 있음.
  - 프로그램 개선이나 시스템 효율화가 대부분 탐색형 문제임.
- **설정형 문제**
  - 미래 상황에 대응하는 문제임.
  - 새로운 과제나 목표를 설정함으로써 일어나는 문제로, 새로운 기능이나 대폭적인 업그레이드가 설정형 문제임.
- 예산과 인력 부족으로 문제점을 한 번에 해결하기 어렵기 때문에 어떤 문제를 선택해 해결했는지 릴리스 노트를 **문제**, **문제점**, **해결책**, **후속 계획** 순으로 적어야 함.
  ```text
  (문제) OOO서비스에 사용자가 급증하면 OO 서버가 정지하는 문제가 있었습니다.
  (문제점) 저희가 파악한 문제점은 잘못된 시스템 설정, 최적화되지 못한 프로그램, 잘못된 DB 설계 등이었습니다.
  (해결책) 그중에서 시스템 재설정으로 OO 서버가 정지하는 문제를 해결했습니다.
  (후속 계획) 보다 안정적인 서비스를 위해 프로그램 최적화와 DB 재설계를 검토하겠습니다.
  ```
- 릴리스 노트에는 면책 조항을 적는 것이 좋은데 **서술어가 중요**함.
  - 행동을 유도할 때 그 행동이 필수인지, 권장인지, 선택인지를 명확히 알려줘야 함.
  - **필수**: 독자가 그 일을 하지 않아서 생기는 문제에 책임을 지지 않음.
    - ex) `~해야 한다`, `~하지 않으면 안 된다`, `~하면 안 된다`, `~해서는 안 된다`
  - **권장**: 어떤 문제를 해결하는 방법이 여러 가지 있는 경우 우선순위를 정해줄 때 씀. 권장하지 않은 방법을 써서 생기는 문제는 책임지지 않음.
    - ex) `~할 것을 권장한다`, `~하는 것이 좋다`, `~하는 것이 이상적이다`
  - **선택**: 여러 옵션을 자유롭게 줄 수 있을 때 사용함. 문제가 되지 않을 내용이나 새로운 기능의 사용을 독촉할 때 씀.
    - ex) `~할 수도 있다`, `~해도 된다`, `~하는 방법이 있다`
- 새로운 기능을 사용할 때 기대한 결과가 나오지 못하는 경우에는 예외 사항을 명시해야 면책이 될 수 있고, 그 예외 사항을 다음 릴리스 노트에서는 해결 대상으로 적기 위해 노력해보는 것이 좋음.

## 04 비즈니스를 이해하는 장애 보고서 쓰기
- **장애 보고서 특징**
  - 개발자가 원할 때 쓸 수 없음. ➡️ 신속한 글쓰기가 필요함.
  - 장애의 1차 원인은 대부분 다른 원인의 결과임. ➡️ 원인과 이유를 찾는 분석적 글쓰기가 필요함.
  - 장애 보고를 받는 윗사람은 대부분 개발자가 아님. ➡️ 상사를 고려한 비즈니스 관점의 글쓰기가 필요함.
  - 장애를 해결했다고 해서 100% 해결한 것이 아님. ➡️ 원하는 것을 얻는 정치적 글쓰기가 필요함.
- **신속한 글쓰기**
  - 대화를 글로 옮기는 방법이 있음.
    > - `무슨 일이야?` - 장애 내용
    > - `그래서 어떻게 됐어?` - 장애 영향
    > - `왜 그렇게 됐어?` - 장애 원인
    > - `넌 어떻게 했는데?` - 조치 사항
    > - `그럼 이제 장애는 해결된 거야?` - 조치 결과
    > - `누가 잘못한 거야?` - 핵심 원인
    > - `그럼 이제 어떻게 할 것인가?` - 향후 대책
- **분석적 글쓰기**
  - **5Whys 기법**: 문제의 원인이 되는 인과 관계를 탐색할 때 Why를 5번 반복해서 질문하는 방식
  - Why는 두 가지 의미를 갖는데 하나는 사물이나 현상이 문제를 초래하는 원인이고, 둘째는 사람이 문제를 초래한 이유임.
  - IT 분야에서는 대부분 사람의 문제로 장애가 재발함. 개발자 사이의 커뮤니케이션 오류가 바로 핵심 원인임.
- **비즈니스 관점의 글쓰기**
  - 장애를 비즈니스 관점으로 쓴다는 것은 매출과 비용 관점으로 설명하는 것임.
  - 장애로 인한 손실을 계산해서 보고해야 함. `매출 손실 = (기대 매출 - 실제 매출 - 지연 매출)`
  - 결제와 관계없는 장애는 `비용 손실 + 기대 매출 손실` 로 구함.
- **정치적 글쓰기**
  - 상사 눈치를 살피면서 쓰는 것이 아니라 정확한 단어와 문장으로 현상과 사실을 있는 그대로 표현하는 것임.
  - 윗사람은 의사결정을 내리기 위한 정확한 정보를 필요로 하기 때문에 수치적으로 확정해서 말하는 것이 필요함.
  
    | 장애 재발 가능성 | 표현 |
    | :---: | :--- |
    | 1% | 절대 재발하지 않는다 |
    | 10% | 재발하지 않는다 |
    | 20% | 재발할지도 모른다 |
    | 30% | 재발할 수도 있다 |
    | 40% | 재발한다고 볼 수도 있다 |
    | 50% | 재발할 수 있다 |
    | 60% | 재발하지 않는다고 볼 수 없다 |
    | 70% | 재발한다고 보여진다 |
    | 80% | 재발한다 |
    | 90% | 재발할 것이 틀림없다 |
    | 99% | 반드시 재발한다 |
